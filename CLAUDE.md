# Pixela Buttons — 仕様書

## プロジェクト概要

[Pixela](https://pixe.la/) のAPIを使って、日々の習慣を記録するスマホアプリ。
Pixelaはユーザーが自由にグラフを定義し、日々の活動量（数値）をピクセルとして記録・可視化できるサービス。

開発者はPixela自身の開発者であり、既存ユーザー向けのクライアントアプリとして位置づける。

---

## 技術スタック

| 項目 | 内容 |
|------|------|
| フレームワーク | Flutter |
| 対象プラットフォーム | iOS / Android（クロスプラットフォーム） |
| 認証情報の保存 | `flutter_secure_storage`（トークン）/ `shared_preferences`（ユーザー名・カード設定） |

### 技術選定の背景

- iOS・Android両対応をひとつのコードベースで実現したいため、クロスプラットフォームフレームワークを採用。
- Flutter は React Native と比較して、初心者にとって環境構築からデバッグまでの体験が整っており挫折しにくい。
- トークンはPixelaの認証に使う機密情報であるため、OSが提供するセキュアな領域（iOS: Keychain / Android: KeyStore）に保存する `flutter_secure_storage` を採用。平文保存は不可。

---

## ファーストリリーススコープ

### 対象外（将来バージョンで追加予定）

- ストップウォッチ機能（状態管理が複雑になるため）
- ホーム画面ウィジェット対応（Flutter単体では実装できずネイティブコードが必要なため）

## 画面構成

```
初回起動画面
　├── 新規登録画面
　└── （認証情報保存後）ホーム画面
　　　└── ボタン追加・編集画面
　　　　　└── グラフ選択画面（APIから取得）
```

## 各画面の仕様

### 1. 初回起動画面

ユーザー名とトークンを入力・保存する画面。アプリ初回起動時のみ表示される。

**入力項目**

| 項目 | 必須 | 備考 |
|------|------|------|
| ユーザー名 | ✓ | |
| トークン | ✓ | 入力内容は伏字・表示切替ボタンあり |

**導線**

- 「アカウントをお持ちでない方はこちら」のリンクから新規登録画面へ遷移する。
  - Pixelaはアカウント登録もAPIで行う仕様のため、アプリ内でサポートする。
  - 既存ユーザーがメインターゲットだが、新規ユーザーの登録もアプリ内で完結できるようにする。

**トークン無効時の挙動**

APIから401エラーが返ってきた場合、保存済みトークンを削除して即座にこの画面へ強制遷移する。
遷移後、画面上部に「トークンが無効になったため、再度設定してください」を一時的に表示する。

- Pixelaではトークンの変更がAPIで可能なため、このケースが発生しうる。
- ダイアログを挟まず即座に遷移するシンプルな設計とした。ただしユーザーが状況を把握できるよう、バナーメッセージで理由を伝える。

---

### 2. 新規登録画面

`POST /v1/users` を使ってPixelaアカウントを作成する画面。

**入力項目**

| 項目 | 必須 | バリデーション |
|------|------|--------------|
| ユーザー名 | ✓ | `[a-z][a-z0-9-]{1,32}` |
| トークン | ✓ | 8〜128文字（`[ -~]`） |
| 利用規約への同意 | ✓ | チェックボックス・利用規約へのリンクを添える |
| 未成年でない（または保護者同意） | ✓ | チェックボックス |
| ~~サンクスコード~~ | 任意 | ファーストリリース対象外 |

**サンクスコードを除外した理由**

Patreonサポーター向けの機能であり、ターゲットが限られる。後から追加するのが容易なため、ファーストリリースはシンプルさを優先して除外した。

---

### 3. ホーム画面

登録済みのカード一覧を表示するメイン画面。

**レイアウト**

- 上部ナビゲーションバー：アプリ名 + 右上に「＋」アイコン
- カード一覧（初回は空）
- 下部タブバー：ホーム / グラフ / 設定

**カードの表示内容**

| 要素 | 内容 |
|------|------|
| ドラッグハンドル | カード左端に6点ドット（並び替え用） |
| emoji | ユーザーが設定したemoji |
| グラフ名 | ユーザーが設定した表示名 |
| 単位・今日の累計値 | 「単位: km　今日: 8km」の形式で表示 |
| ボタン列 | 固定値ボタン（横スクロール対応）＋「カスタム」ボタン |
| ↗ アイコン | グラフHTMLを外部ブラウザで開く |
| ⋮ アイコン | 編集・削除メニュー |

**カードの操作**

| 操作 | アクション |
|------|-----------|
| 固定値ボタンをタップ | add / subtract APIを実行 → ダイアログ表示 |
| 「カスタム」ボタンをタップ | 数値入力ダイアログを表示 → add / subtract APIを実行 → ダイアログ表示 |
| 長押し → ドラッグ | 並び替え（`ReorderableListView`） |
| ↗ アイコンをタップ | グラフHTMLを外部ブラウザで開く |
| ⋮ アイコンをタップ | 編集・削除メニューを表示 |

**ボタン列の設計背景**

- `increment` エンドポイントは+1固定のため、任意の値を加算・減算できる `add` / `subtract` エンドポイントを採用。
- 毎回数値入力を求めるのは使い勝手が悪いため、よく使う値を「固定値ボタン」として事前登録できる設計にした。
- 固定値ボタンの個数制限はなし。ボタンが増えた場合は横スクロールで対応する。
- 「カスタム」ボタンは全カードに自動で固定表示される補助機能。正の数を入力した場合はadd、負の数を入力した場合はsubtract（絶対値を渡す）として処理する。これによりユーザーが「加算か減算か」を意識しなくて済む。
- 並び替えは長押し → ドラッグに専念させ、編集・削除は ⋮ メニューに分離した。これにより「長押し」の役割が明確になりUXの混乱を避けられる。

**ボタンの色設計**

- 加算ボタン：ユーザーが指定した色
- 減算ボタン：加算色を暗くしたトーンを自動生成（同系色で統一感を保つ）
- カスタムボタン：全カード共通・ボーダーのみのニュートラルな色
- 補色案も検討したが、色の組み合わせによっては読みにくくなるケースがあるため暗いトーン案を採用した。

---

### 4. ボタン追加・編集画面

カードに表示するボタンの設定を行う画面。ホーム画面の「＋」から遷移する。

**設定項目**

| 項目 | 内容 |
|------|------|
| グラフ | APIから取得したグラフ一覧から選択。graph-idと単位をサブテキストで表示 |
| 表示名 | 任意入力（初期値はグラフ名） |
| emoji | 任意設定 |
| ボタン色 | 加算ボタンの色を指定。減算ボタン色は自動生成 |
| 固定値ボタン | 値を登録・並び替え・削除できるリスト。右上「＋」で追加 |

**設計背景**

- グラフ一覧ありきではなく「ボタン一覧ありき」の設計にしたのは、ユーザーが「何を記録したいか」という目的から操作を始められるため自然なUXになるから。
- グラフ一覧はAPIから取得して選択させる（手入力ではなく）。

## 記録後ダイアログの仕様

ボタンタップ後のフロー：

```
ボタンタップ
　↓
add / subtract APIを叩く（isRejectedならリトライ、詳細は下記）
　↓（成功）
/today APIを叩く
　↓
【成功】                       【/today 失敗】
「+3km を記録しました」         「+3km を記録しました」
「今日の合計: 11km」            「累計値の取得に失敗しました」
```

**ダイアログの閉じ方**

- 「OK」ボタンをタップして閉じる
- 数秒後に自動で閉じる（どちらでも閉じられる）

## APIリクエストの仕様

### エラーハンドリング

| 条件 | 挙動 |
|------|------|
| 503 + `isRejected: true` | 成功するまで無限リトライ（500ms間隔） |
| 401 Unauthorized | リトライなし・トークン削除・初回起動画面へ強制遷移 |
| その他のエラー | リトライなし・ユーザーにエラー表示 |

**リトライ設計の背景**

Pixelaには、Pixela Supporterでないユーザーに対して一部のAPIリクエストを25%の確率でリジェクトする仕様がある（503 + `isRejected: true`）。このアプリを使う特典として、このリジェクトを透過的にリトライすることでユーザーに意識させない設計にした。リジェクトは一時的なものであり必ずいつかは成功するため、このケースに限り無限リトライとする。ネットワーク断など他の原因による失敗はリトライしない。

全APIリクエストのインターセプターで401と503を一元的に捕捉する実装が望ましい（`dio` パッケージのインターセプター機能を推奨）。

## カードデータの保存形式

カード・固定値ボタンの設定は `shared_preferences` にJSON形式でシリアライズして保存する。

```json
[
  {
    "id": "uuid",
    "graphId": "running",
    "displayName": "ランニング",
    "emoji": "🏃",
    "color": "#1D9E75",
    "unit": "km",
    "buttons": [
      { "id": "uuid", "value": 3 },
      { "id": "uuid", "value": 5 },
      { "id": "uuid", "value": -3 }
    ]
  }
]
```

- `value` が正の数 → `add` エンドポイント
- `value` が負の数 → `subtract` エンドポイント（絶対値を渡す）
- カードの順序はリストの順序で管理する
